# PortFree 开发计划书

## 1. 项目概述

**项目名称：** PortFree  
**平台：** macOS  
**技术方向：** SwiftUI + AppKit 能力补充 + 系统命令调用  
**核心目标：** 让开发者可以快速识别、定位并结束占用指定端口的进程，同时提供常见的端口管理辅助功能。

PortFree 面向本地开发场景，重点解决以下痛点：
- 启动服务时报“端口已被占用”，但不想手动执行 `lsof`、`netstat`、`kill`。
- 同一个端口经常被不同进程占用，需要更快地查看和结束。
- 需要保留常用端口、最近操作记录、批量清理等辅助能力。

---

## 2. 产品定位

PortFree 是一个轻量级 macOS 桌面工具，不做复杂系统管理平台，优先聚焦“快速释放端口”这一高频核心需求。

### 目标用户
- 本地开发者（前端、后端、全栈、移动开发）
- 经常运行本地服务的测试/运维人员
- 使用 `3000`、`8080`、`5173`、`8000`、`9000` 等开发端口的用户

### 产品价值
- 降低命令行操作门槛
- 减少排查端口占用的时间成本
- 提供更直观、更安全的杀进程体验

---

## 3. 需求背景与问题定义

### 当前常见流程
1. 用户发现端口启动失败
2. 打开终端执行 `lsof -i :端口号`
3. 找到 PID
4. 再执行 `kill -9 PID` 或 `kill PID`
5. 如果失败，继续排查权限、子进程、重复端口占用等问题

### 当前流程存在的问题
- 操作分散，效率低
- 对非命令行熟练用户不友好
- 容易误杀错误进程
- 缺少历史记录和常用端口管理

### PortFree 要解决的问题
- 能否在 3 秒内完成“查询端口 + 结束进程”操作
- 能否让用户明确知道“这个端口被谁占用”
- 能否在保证效率的同时，降低误操作风险

---

## 4. 产品目标

### 核心目标（MVP）
- 输入端口号后，快速查询占用进程
- 一键结束占用该端口的进程
- 显示基础进程信息，帮助用户确认是否应结束
- 提供明确的成功/失败反馈

### 二期目标
- 常用端口收藏
- 最近清理记录
- 批量扫描常见开发端口
- 菜单栏快速操作

### 非目标
- 不做系统级进程监控平台
- 不做完整防火墙/网络诊断工具
- 不做远程服务器端口管理

---

## 5. 功能范围

## 5.1 MVP 功能

### 1）端口查询
用户输入端口号后，应用查询该端口当前是否被占用。

**展示信息建议：**
- 端口号
- 占用状态（空闲 / 已占用）
- 进程名
- PID
- 协议类型（TCP / UDP，如可获取）
- 启动路径或命令片段（如可获取）

### 2）结束进程
当端口被占用时，用户可点击按钮结束该端口对应进程。

**建议策略：**
- 默认先尝试温和结束（`kill PID`）
- 若失败，再提供“强制结束”（`kill -9 PID`）
- UI 中明确区分“结束进程”和“强制结束”

### 3）结果反馈
操作完成后展示结果：
- 已成功释放端口
- 结束失败
- 无权限
- 端口未被占用
- 查询异常

### 4）输入校验
- 仅允许输入有效端口范围 `1...65535`
- 空值、非数字、超范围时应即时提示

### 5）基础历史记录
记录最近执行过的查询和清理动作。

**字段建议：**
- 时间
- 端口号
- PID
- 进程名
- 动作类型（查询 / 结束 / 强制结束）
- 结果状态

## 5.2 增强功能（V1.1 / V1.2）

### 1）常用端口收藏
用户可以收藏常见端口，点击即可快速检查或清理。

**预设建议：**
- 3000
- 5173
- 8080
- 8000
- 8081
- 9000

### 2）批量扫描常用端口
一键扫描多个常见开发端口，展示哪些被占用。

### 3）菜单栏模式
在 macOS 菜单栏常驻，支持：
- 快速输入端口查询
- 查看最近占用端口
- 一键释放收藏端口

### 4）复制命令
提供“复制终端命令”能力，例如复制 `lsof -i :3000`、`kill -9 1234`。

### 5）详情页
点击占用结果后展示更多信息：
- 完整命令路径
- 所属用户
- 父进程信息（如可获取）
- 最近操作记录

## 7. 典型使用流程

### 场景 A：快速释放单个端口
1. 用户打开 PortFree
2. 输入端口 `3000`
3. 应用显示该端口被某进程占用
4. 用户点击“结束进程”
5. 应用反馈“端口已释放”

### 场景 B：温和结束失败，改为强制结束
1. 用户查询端口
2. 点击“结束进程”失败
3. 应用提示可尝试“强制结束”
4. 用户二次确认后执行强制结束
5. 应用更新结果

### 场景 C：查看近期记录并重复操作
1. 用户打开历史记录
2. 看到最近被占用的 `5173`
3. 点击该记录重新查询
4. 若仍被占用则再次清理

## 9. UI/交互建议

### 首页结构建议
- 顶部：应用标题 + 简短说明
- 中间：端口输入框 + 查询按钮
- 查询结果区：
  - 端口状态
  - 进程信息
  - 操作按钮（结束 / 强制结束）
- 侧边或下方：
  - 常用端口
  - 最近记录

### 交互原则
- 单窗口即可完成 MVP
- 操作路径尽量不超过 2 次点击
- 危险操作必须明确提示
- 结果反馈即时且醒目

### 文案风格建议
- 清晰直接，避免系统化术语过多
- 示例：
  - “端口 3000 当前空闲”
  - “发现占用进程：node (PID 1234)”
  - “已成功结束进程并释放端口”
  - “普通结束失败，可尝试强制结束”

## 10. 技术方案建议

## 10.1 技术栈
- **界面层：** SwiftUI
- **系统能力补充：** 必要时结合 AppKit
- **数据持久化：** SwiftData（适合历史记录、收藏端口）
- **命令执行：** `Process` + `Pipe`
- **权限与系统交互：** 基于 macOS 沙盒限制评估实现路径

## 10.2 核心实现思路

### 端口查询
建议优先通过系统命令实现：
- `lsof -i :<port>`
- 或 `lsof -nP -iTCP:<port> -sTCP:LISTEN`

然后在应用中解析输出，提取：
- processName
- pid
- user
- protocol
- port
- command

### 杀进程
通过系统命令执行：
- 普通结束：`kill <pid>`
- 强制结束：`kill -9 <pid>`

### 数据模型建议
可定义以下实体：

**PortCheckRecord**
- id
- port
- processName
- pid
- actionType
- resultStatus
- createdAt

**FavoritePort**
- id
- port
- aliasName
- createdAt

### 模块拆分建议
- `PortQueryService`：负责端口查询
- `ProcessKillService`：负责结束进程
- `CommandRunner`：负责执行 shell 命令
- `HistoryStore`：负责历史记录持久化
- `FavoritePortStore`：负责收藏端口持久化
- `ContentViewModel`：负责页面状态管理

## 11. 风险与约束

### 1）macOS 权限问题
部分进程可能无法直接结束，尤其是系统进程或非当前用户进程。

**应对策略：**
- MVP 先支持当前用户可操作的进程
- 失败时明确提示权限不足
- 后续再评估提权/辅助工具方案

### 2）App Sandbox 限制
如果未来计划上架 Mac App Store，需要重点评估 shell 命令执行和系统信息读取限制。

**建议：**
- 先以开发者工具方向推进，优先跑通非商店分发版本
- 后续再评估沙盒兼容策略

### 3）误杀风险
用户可能结束错误进程，导致本地服务中断。

**应对策略：**
- 显示清晰的进程名和 PID
- 强制结束前增加确认
- 默认先提供普通结束

### 4）命令输出兼容性
不同 macOS 版本下 `lsof` 输出格式可能存在细微差异。

**应对策略：**
- 为解析层设计健壮的容错逻辑
- 对异常输出做兜底显示

## 13. 验收标准

### MVP 验收标准
- 用户可以输入合法端口并发起查询
- 应用能正确识别端口是否被占用
- 占用时能展示 PID 与进程名
- 用户可执行普通结束和强制结束
- 操作结果有清晰反馈
- 失败场景有明确提示

### 体验验收标准
- 常见流程在单窗口内完成
- 从输入端口到执行释放不超过 2 次主要点击
- 文案清晰，无明显歧义
- 对异常场景不会崩溃

## 17. 总结

PortFree 的最佳切入点是做一个**轻量、直接、单用途但高频实用**的 macOS 开发者工具。第一阶段不追求大而全，而是把“查询端口占用 + 一键释放”做到足够快、足够准、足够稳。在 MVP 跑通后，再逐步增加收藏端口、历史记录和菜单栏能力。
